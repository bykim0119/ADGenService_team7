## 1. validate란 무엇인가

### 한 줄 정의

> 사용자가 올린 입력이 "처리 가능한 입력인지" 먼저 확인하는 단계.

### 왜 필요한가

에러가 났을 때 원인은 항상 둘 중 하나다.

| 원인 | 예시 |
|------|------|
| **입력 문제** | 파일이 깨짐 / 지원하지 않는 형식 / 이미지가 너무 이상함 |
| **시스템 문제** | API 실패 / 모델 서버 지연 / worker 오류 / 일시적 timeout |

이 둘을 구분하지 않으면 사용자 입장에서는 둘 다 그냥 "실패"로 보인다.  
그러면 뭘 다시 올려야 하는지, 그냥 다시 시도하면 되는지, 시스템이 고장난 건지 알 수가 없다.

### validate의 핵심 철학

validate는 "무조건 막는 단계"가 아니다.

1. 잘못된 입력은 구분해서 알려주고
2. 시스템 실패는 시스템 실패로 따로 다루고
3. 애매한 경우에는 입구를 괜히 막지 않도록 fallback도 고려

즉 validate의 목적은 **사용자를 괴롭히는 검사가 아니라, 문제 원인을 깨끗하게 나누는 것**이다.

### fallback이란

원래 하려던 검증이 신뢰성이 낮거나 실패할 때, 더 단순한 방법으로 대체하는 것.  
예: AI로 "음식 사진인지" 판단하려 했으나 판정 신뢰성이 낮아서 → 업로드 성공 시 무조건 통과.  
초기 통합 단계의 안정성 우선 fallback이며, 이후 판단 로직이 안정되면 교체 가능.

## 2. 4가지 통합 포인트

### 1) 업로드 정규화

**무엇인가**  
사용자가 올린 이미지를 AI가 안정적으로 처리할 수 있는 형태로 미리 변환하는 것.

**왜 필요한가**  
- 아이폰 등 모바일 사진은 EXIF 회전 정보가 있어서 그냥 넣으면 옆으로 누운 상태로 처리됨  
- PNG/JPG가 아닌 포맷(HEIC 등)은 일부 파이프라인에서 에러 발생  
- RGB가 아닌 모드(RGBA, CMYK 등)는 OpenAI image edit API에서 오류 발생 가능  

**처리 순서**

```text
PIL.Image.open(BytesIO(file_bytes))
→ ImageOps.exif_transpose(image)
→ image.convert("RGB")
→ image.save(path, format="PNG")
```

**팀 repo 적용 포인트**  
`backend/main.py`의 `/generate` 엔드포인트에서 `product_image` 업로드를 받는 부분.  
현재는 raw bytes를 base64로 변환해서 그대로 넘기는 구조 → 정규화 삽입 필요.

### 2) validate 분기 기준

**무엇인가**  
문제가 생겼을 때 입력 문제인지 시스템 문제인지 나눠서 처리하는 기준.

**분기 기준**

| 상황 | 분류 | UX 처리 |
|------|------|---------|
| 파일 형식 불가 | 입력 문제 | "다른 형식의 파일을 올려주세요" |
| 이미지 파싱 실패 | 입력 문제 | "파일이 손상된 것 같아요" |
| API 호출 실패 | 시스템 문제 | "잠시 후 다시 시도해주세요" |
| worker 오류 | 시스템 문제 | "서버 오류가 발생했어요" |
| 판단 불가 (애매함) | fallback | 통과 처리 후 생성 진행 |

**팀 repo 적용 포인트**  
`backend/main.py` 업로드 처리 전 단계, 또는 Celery task 내부 예외 처리 분기.

### 3) generating/result 소비 흐름 안정화

**무엇인가**  
job_id 기반 비동기 구조에서 프론트가 언제 완료로 보고, 결과를 어떻게 소비할지 기준.

**핵심 문제**  
- 폴링을 언제 멈출지 기준이 없으면 프론트가 무한 대기하거나 너무 빨리 끊음  
- 결과를 다시 조회할 때 재생성이 일어나면 비용/시간 낭비  
- 실패 원인을 상태에 담지 않으면 프론트가 에러를 구분할 수 없음  

**권장 상태 구조**

| status 값 | 의미 | 프론트 처리 |
|-----------|------|------------|
| `queued` | 작업 대기 중 | 로딩 표시 |
| `processing` | 생성 중 | 로딩 + 진행 표시 |
| `completed` | 완료 | 결과 화면으로 이동 |
| `failed_input` | 입력 문제로 실패 | 입력 수정 안내 |
| `failed_system` | 시스템 문제로 실패 | 재시도 안내 |

**result 재진입 캐시**  
동일 job_id로 result를 다시 조회할 때 재생성 없이 캐시된 결과를 반환하는 구조.  
인메모리 dict 또는 Redis 활용 가능 (팀 repo는 이미 Redis 있음).

---

### 4) 결과물 저장/구분 기준

**무엇인가**  
AI가 만든 결과물을 단계별로 구분해서 저장하고 소비하는 기준.

**결과물 3단계**

| 단계 | 내용 | 저장 여부 |
|------|------|----------|
| `raw` | AI가 생성한 원본 이미지 | 선택적 저장 |
| `composited` | 누끼 + 배경 합성 결과 | 저장 권장 |
| `final_export` | 텍스트 오버레이 + 비율 적용 최종본 | 반드시 저장 |

**Supabase 적용 방향**  
현재 `assets` 테이블 `generated_image_url` 단일 구조 기준.  
향후 스키마 확장 후보: `raw_image_url` / `composited_image_url` / `final_image_url` 분리.  
최소 적용 시 `composited`와 `final_export` 구분만으로도 UX 개선 효과 있음.
